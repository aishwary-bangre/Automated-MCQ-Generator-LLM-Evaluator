# Final Project Problem Statement: AI-Powered Benchmarking & Content Generation Dashboard

## 1. Overview
The objective is to build an automated, interactive educational content pipeline and web dashboard. This platform leverages Retrieval-Augmented Generation (RAG) to anchor generated questions in official NCERT textbook chapters. It utilizes a multi-agent framework to engineer realistic diagnostic distractors based on student misconceptions, and runs a programmatic benchmarking suite to evaluate local open-source models versus commercial cloud models.

## 2. Core Objectives
* Benchmarking Performance: Evaluate and compare LLMs based on inference latency, token efficiency, and AI-judged accuracy scores.
* Content Generation: Generate high-quality mathematics MCQs for Class 10, strictly grounded in the official curriculum.
* Interactive Evaluation: Provide a user-friendly Streamlit UI to interact with generated content and directly compare local (Qwen) vs cloud (Gemini) generation side-by-side.

## 3. Architecture & Pipeline
### A. Data Ingestion & Retrieval (RAG)
* Target Scope: Class 10 Mathematics (Quadratic Equations, Arithmetic Progressions, Coordinate Geometry, Probability).
* Document Processing: Parse NCERT Math PDFs to preserve text and complex LaTeX formulas, chunking by paragraphs.
* Vector Storage: Generate embeddings using `sentence-transformers` (all-MiniLM-L6-v2) and store them in a local ChromaDB instance.
* Retrieval: Perform semantic search to retrieve the most relevant mathematical formulas and concepts based on user-selected sub-topics.

### B. Dual-Model Execution Engine
* Parallel Pipelines: Execute independent generation paths for a local model (`Qwen2.5-0.5B-Instruct` via HuggingFace) and a cloud API (`Gemini 3.5 Flash` via Google GenAI SDK).
* Agentic Workflow: Each model acts as two specialized agents:
    * Agent 1 (The Educator): Generates one Question, the Correct Answer, and a Step-by-step Explanation based solely on retrieved context.
    * Agent 2 (The Student Misconception Specialist): Analyzes the correct pair and generates exactly three logical incorrect options (distractors) modeled on verified student errors.
* Structured Output: Enforce deterministic JSON schemas using custom prompting, fault-tolerance, and regex sanitizers to handle escape characters.

### C. Benchmarking Suite
* Metrics Collection: Record inference latency and token usage per generation.
* LLM-as-a-Judge Evaluator: Use the Groq API (`Llama-3.3-70b-versatile`) to programmatically judge both generated outputs.
* Scoring Criteria: Evaluates on a 1-10 scale for Hallucination, Logic, and Plausibility of Distractors.

## 4. User Interface (Streamlit Dashboard)
* Syllabus Mapper: Users select the overarching math chapter and a highly specific sub-topic to generate questions for.
* Interactive MCQ Interface: Displays generated MCQs and explanations side-by-side for direct comparison. Includes graceful fallback UI for API timeouts or local OOM errors.
* Performance Dashboard: Visual metrics displaying latency, hardware tokens, and LLM-as-a-Judge evaluation scores.

## 5. Success Criteria & Constraints
* Edge Hardware Execution: The local model (`Qwen2.5-0.5B-Instruct`) must run efficiently inside the constraints of a 4GB VRAM GPU (GTX 1650) and limited System RAM (1GB) using 4-bit quantization or strict precision management.
* Production Integrity: 0% hallucination rate enforced by strictly anchoring generation to the retrieved textbook context.
* Frameworks: Built strictly using Python, ChromaDB, HuggingFace transformers, Google GenAI SDK, Groq API, and Streamlit.
